## 系统配置

| 项目 | 详情 |

## 问题

macOS 下完全没有任何音频输出设备。系统只显示 BlackHole 虚拟设备，所有物理音频接口（3.5mm 前后面板）均不可用。

---

## 尝试过的方案

### 1. AppleALC（标准版）+ layout-id 逐个测试

**原理**：AppleALC 通过注入 PinConfigurations 让 AppleHDA 驱动未受支持的 codec。

**尝试的 layout-id**：1, 7, 11, 12, 13, 21, 22, 23, 31, 66, 69, 77, 98, 99（全部 12 个官方支持 layout）

**结果**：❌ 全部失败。AppleALC 能加载，`alc-layout-id` 能注入到 IORegistry，但 **PinConfigurations 始终为 `<00>`**，无任何 AppleHDAEngine 创建。

**根因分析**：ALC897 codec revision (0x00100402) 与 AppleALC 内置的引脚配置数据不匹配。这不是 layout 选择的问题，而是 AppleALC 的 codec revision 匹配逻辑在 Tiger Lake 平台上无法找到对应条目。

### 2. HDA 控制器 device-id 欺骗

**原理**：Tiger Lake HDA 控制器 (8086:43C8) 不被 macOS 原生支持，需要伪装成兼容的控制器。

**尝试的欺骗目标**：

| device-id | 对应平台 | 结果 |
|-----------|----------|------|
| C8340000 (0x34C8) | Ice Lake-LP (InsanelyMac 官方推荐) | ❌ PinConfigurations 为空 |
| C8060000 (0x06C8) | 400 系 PCH (社区推荐 ALC897+500系) | ❌ PinConfigurations 为空 |

两种欺骗都能让 AppleHDAController 成功挂载，但 AppleALC 的引脚配置始终无法应用。

### 3. AppleALCU（统一版/数字专用版）

**原理**：AppleALCU 是 AppleALC 的精简变体，尝试用不同的匹配逻辑。

**结果**：⚠️ 能创建音频设备，但**只提供 HDMI/DP 数字音频**，不含模拟音频（3.5mm 插孔）。AppleALCU 砍掉了所有模拟音频补丁逻辑。

**教训**：AppleALCU ≠ AppleALC，它是数字音频专用版，不适合需要模拟输出的场景。

### 4. codec revision 欺骗

**原理**：通过 DeviceProperties 注入 `alc-revision` 将 codec 版本伪装成 AppleALC 支持的版本。

**尝试**：`alc-revision = 01001000`（伪装为 0x100001，AppleALC 最常见的默认 revision）

**结果**：❌ `alc-revision` 成功注入 IORegistry，但 PinConfigurations 仍为空。Revision 欺骗对 Tiger Lake 平台无效。

### 5. 辅助启动参数

尝试过以下 boot-args 组合：
- `alcid=12` / `alcid=66` — 直接指定 layout-id
- `-alcbeta` — 启用 AppleALC 实验性 layout
- `alcdelay=1000` — 延迟初始化（等待硬件就绪）
- `alctsel=1` — 解决 Windows 热重启后音频失效

**结果**：❌ 均无效。

### 6. 手动 PinConfigurations 注入

**原理**：通过 DeviceProperties 直接注入引脚配置数据，绕过 AppleALC 的自动匹配。

**结果**：❌ 数据出现在 IORegistry 但 AppleHDA 不创建音频引擎。手动注入的引脚配置与 AppleHDA 的内部匹配逻辑不兼容。

### 7. VoodooHDA（通用 HDA 驱动）

**原理**：VoodooHDA 直接探测 HDA 硬件，不依赖 AppleALC/layout-id/AppleHDA。

**尝试的修复**：
- 移除损坏的 `_CodeSignature`
- Block AppleHDAController 避免冲突
- 多次开关启用/禁用

**结果**：❌ **VoodooHDA 从未被 OpenCore 注入**。启动日志中没有任何 "Voodoo" 字样。这是整个过程中最大的未解之谜——kext 存在、配置正确、签名已清除，但 OC 就是不加载它。可能的原因：OC 与 ERYING 固件的某些交互导致 kext 注入跳过。

### 8. ESP 文件系统修复

**遇到的问题**：ONDA 硬盘的 ESP (disk1s1) 文件系统不可靠，配置修改后频繁丢失。

**修复**：`diskutil eraseVolume fat32 EFI /dev/rdisk1s1` 格式化后 rsync 完整 EFI 目录。

**结果**：✅ 文件持久性修复。

### 9. ONDA 启动引导修复

**说明**：最初系统从 USB 盘启动 OC，拔掉 USB 后无法启动。修复 ONDA ESP 后实现了独立启动。

**结果**：✅ 系统可从 ONDA 硬盘独立启动。

---

## 最终解决方案

### ✅ USB 声卡（免驱）

macOS 内置 `AppleUSBAudio` 驱动，绝大多数 USB 音频设备即插即用，无需任何配置。

插入 USB 声卡后，系统自动识别：
- 输入设备：USB 声卡麦克风输入
- 输出设备：USB 声卡耳机/音箱输出

不需要修改任何 OpenCore 配置或 kext。

## 蓝牙修复记录 (2026-06-25)

### 问题诊断

- **Intel AX201 Bluetooth (0x8087:0x0029)** USB 设备已被识别
- `IntelBluetoothFirmware` (v2.2.0) 匹配设备但 **固件上传失败**：`fw_name=""`, `!registered, !matched`
- `IntelBTPatcher` (v2.2.0) 也是 `!registered, !matched`
- 根因：v2.2.0 固件选择逻辑不支持 bcdDevice=1 的此款芯片

### 修复操作

1. **更新 IntelBluetoothFirmware**: v2.2.0 → v2.4.0（固件选择逻辑和固件库大幅更新）
2. **更新 IntelBTPatcher**: v2.2.0 → v2.4.0
3. **修复 kext 加载顺序**: BlueToolFixup → IntelBTPatcher → IntelBluetoothFirmware
4. **禁用 VoodooHDA**（USB 声卡已工作，不需浪费启动时间）
5. **BlueToolFixup** 保持 v2.6.4（正常运行中）

### 待观察

- USB 端口类型为 0 (USB 2.0)，应为 255 (Internal)。如果固件更新后蓝牙仍不行，需修复 USB 端口映射。

## 关键经验教训

1. **AppleALC 不是万能的**——Tiger Lake 移动平台 + ALC897 + HM570 的组合超出了 AppleALC 当前的支持范围。codec revision 匹配是硬性条件，无法通过 layout-id 切换绕开。

2. **AppleALCU ≠ AppleALC**——AppleALCU 是纯数字音频版本，不能用它驱动模拟输出。

3. **device-id 欺骗有前提**——HDA 控制器欺骗能让 AppleHDAController 挂载，但如果 codec revision 不匹配，后面的一切都不会发生。

4. **VoodooHDA 注入失败是持久谜题**——在 ERYING 这块板子上，OpenCore 就是不注入 VoodooHDA。可能是固件层面的某种保护或兼容性问题。

5. **ESP 文件系统不可靠**——FAT32 ESP 在某些盘上会有静默写入失败的问题。格式化是最快最可靠的修复。

6. **USB 声卡是黑苹果音频的终极兜底方案**——便宜、免驱、音质够用、零配置。
